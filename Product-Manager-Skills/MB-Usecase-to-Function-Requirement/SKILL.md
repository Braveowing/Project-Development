# UC-to-FR: 从 Use Case 拆解 Function Requirement

将 1.5 章节的 UC 需求系统化拆解为 1.7.2 章节的功能需求，保证每条 1.5 需求都有 1.7.2 承接。

## ⚠️ 全局运行原则（最高优先级）

1. **仅操作指定范围**：本 skill 只对任务来源（用户、工作流上游、需求方等）明确指定的 1.5 章节 UC/N-UC 条目进行拆解操作。未指定的条目仅作参考，不纳入追溯矩阵，不产出对应 FR。
2. **支持增量输入**：本 skill 支持对任务来源直接输入的单条或多条 UC/N-UC 条目进行拆解操作，无需依赖完整的 1.5 章节文档。

---

## 拆解原则

1. **完整覆盖**：任务来源指定的每条 UC 需求中，type 为 `requirement` 的条目，必须被至少一条 1.7.2 FR 承接。非 requirement 类型（如 remark、note、example、TBD）的条目视情况参考，不强求承接。
2. **双向可追溯**：每条 FR 必须标注其溯源的 1.5 章节 ID；追溯矩阵只检查任务来源明确指定的 UC 条目。
3. **FR 比 UC 更具体**：FR 需在 UC 基础上充实和细化场景（包括任务来源额外导入的场景）、预期功能表现、预期系统执行逻辑、前置条件、异常处理等实现层面细节。
4. **按功能流程组织**：FR 参考 UC 的场景来具体设计功能，并按功能流程（而非 UC 编号顺序）排列组织。涉及多域切换的场景（如 Park2Park 中的行泊车切换）需在对应章节中体现切换流程。

---

## 拆解流程（关键询问节点）

### 阶段一：输入确认

在开始拆解前，确认以下信息：

1. **确认操作范围**：明确任务来源指定的 1.5 章节 UC/N-UC 条目（章节编号列表或直接粘贴的条目）
2. **场景补充询问**：对于每个 UC，1.5 中通常只覆盖了典型场景。实际调研中可能发现更多泛化的场景类型，而系统在同一个 UC 下应对不同场景的功能要求可能不同。
   - ⛔ **必须询问任务来源**：是否有额外场景需要导入？
   - 得到确切回复后，再进入下一阶段

### 阶段二：拆解执行

对每条指定的 UC 需求执行拆解（详见下方拆解步骤）。

### 阶段三：信号交互确认

在 FR 写作过程中，涉及信号交互时序的部分：
- ⛔ **必须询问任务来源**：是否有现成的信号交互时序图/文档需要导入？
- 得到确切回复后，再完成最终 FR 输出

## 1.5 需求分类与拆解策略

### Scenario 需求类型识别

| 类型 | 特征 | FR 拆解方向 |
|------|------|------------|
| **状态转换** | 「shall set/start/stop/end/activate」 | 补充状态机定义、转换条件、信号代号 |
| **条件检查** | 「shall check/verify/ensure」 | 补充具体检查项、信号来源、失败处理 |
| **能力声明** | 「shall provide/support/detect」 | 补充触发条件、数据来源、输出信号代号 |
| **约束条件** | 「shall not / shall only when」 | 补充安全约束、边界值、异常路径 |
| **HMI 交互** | 「shall display/indicate/inform」 | 补充 HMI 交互逻辑、状态机、超时处理 |
| **数据操作** | 「shall store/save/delete/record」 | 补充数据结构、存储条件、容量限制、覆盖策略 |
| **场景细化** | UC 只覆盖典型场景，需充实更泛化的场景类型 | 识别不同场景下的差异化功能要求，分别拆解 FR |

### 从 UC 到 FR 的拆解步骤

对每条 1.5 需求：

1. **提取核心动作** — 需求要求系统做什么
2. **细化场景** — 将 UC 中的典型场景扩展为完整场景集合（含任务来源额外导入的场景），识别不同场景下的差异化功能要求
3. **补充前置条件** — 触发此需求需要满足什么（用信号代号表示，如 `SIG_FUNC_STATE == READY`，不要求写具体信号值）
4. **定义行为细节** — 具体怎么做（状态转换、信号交互逻辑、预期功能表现、预期系统执行逻辑）
5. **处理异常路径** — 条件不满足时怎么办（降级、提示、重试）
6. **标注信号交互** — 涉及哪些跨系统信号交互，用代号表示逻辑即可（详见阶段三信号确认）
7. **写入 FR** — 用标准 shall 格式

## FR 写作规范

### FR 分配原则

1. **一对一为主**：一条 UC 拆解为一条或少数几条 FR，每条 FR 只承接与其直接相关的 UC
2. **场景不堆叠**：不要在单条 FR 中罗列所有场景和行为。将不同场景按 UC 来源拆分到不同 FR，确保每条 FR 精炼聚焦
3. **精炼输出**：单条 FR 控制篇幅，只描述与该 UC 直接相关的条件、场景、行为和异常

### FR 格式模板

FR 不强制使用 "The system shall..." 句式，可采用总括+分述结构：

```
FR_ID: [功能域-序号]
追溯: 1.5.x.y / 1.5.x.y.z

[一句话总括：概括该 FR 的核心需求，融合前置条件、主要场景、预期行为和异常处理的要点]

前置条件:
- [条件1]: SIG_xxx = [state]
- [条件2]: STATE_MACHINE = [state]

场景:
- [场景A]: [描述]
  - 预期功能表现: [具体表现]
  - 预期系统执行: [系统怎么做]

行为:
- [步骤1]
- [步骤2]
- 性能边界: [涉及性能的量化指标]

信号交互:
- 输入/输出: SIG_xxx (来源/目标系统)
- 时序: [步骤化描述]
（注：具体信号值及时序图需确认任务来源是否有导入）

异常处理:
- [异常情况] → [处理方式]
```

**写作要点**：
- 总括段可自由组织语言，不需要 shall 句式，重点是清晰传达这条 FR 要解决什么
- 场景只列与该 UC 直接相关的，不把其他 UC 的场景混入
- 如一条 UC 拆出多条 FR，每条 FR 各自覆盖该 UC 的不同方面

### 状态机约定

⛔ **首先询问任务来源**：是否有现成的状态机定义需要导入？

- **有现成状态机**：直接引用导入的状态机定义，沿用其状态名和转换逻辑
- **无现成状态机**：根据实际功能需求设计状态机，定义具体状态名。设计原则：
  - 状态名应能清晰表达物理含义（如 `LEARNING_READY`、`CRUISING_ACTIVE`）
  - 每个状态需定义进入条件、退出条件、该状态下允许的行为
  - 状态转换需标注触发条件（信号代号或事件）

### 参数引用格式

参数使用 `Parameter_Name (value, type)` 格式引用：
- `Max_Speed_Cruising (18 kph, SCN)` — 诊断编码参数
- `Max_Num_Slot (2000, calibratable)` — 可标定参数
- `safety_distance_lat (0.10 m, PRM)` — 编译时静态参数

## 组织结构：1.7.2 章节模板

FR 按功能流程组织（非 UC 编号顺序）。以下为参考章节，可**根据实际 UC 情况灵活匹配或新增**：

```
1.7.2 Functional requirements
  1.7.2.1  Initialize [功能名]
         → 通用初始化、登录检查、编码检查、系统就绪检查

  1.7.2.2  [功能名] User Consent
         → 用户同意/授权管理

  1.7.2.3  [功能名] Learning
    1.7.2.3.1  Detect ODD             → ODD 检测与自动学习启动
    1.7.2.3.2  Stop and restart        → 学习暂停与恢复
    1.7.2.3.3  Set new recording Map   → 新建地图 ID
    1.7.2.3.4  Set Park-Position       → 设置车位（Park like APA / Park like I did）
    1.7.2.3.5  Set Exit Position       → 设置出口
    1.7.2.3.6  Record Corridor         → 记录走廊（结构化/非结构化）

  1.7.2.4  [功能名] Map Manager
    1.7.2.4.1  Recorded map list       → 地图列表查询
    1.7.2.4.2  Create map              → 创建地图
    1.7.2.4.3  Check and select        → 检查与选择
    1.7.2.4.4  Delete                  → 删除
    1.7.2.4.5  Rename                  → 重命名
    1.7.2.4.6  Synthetic MAP View      → 合成地图渲染

  1.7.2.5  [功能名] Pre-Start
         → 启动前条件检查、定位、就绪判定

  1.7.2.6  [功能名] Maneuvering
    1.7.2.6.1  Route planning          → 路径规划与导航
    1.7.2.6.2  Obstacle handling       → 障碍物处理（绕行/倒车/等待）
    1.7.2.6.3  User intervention       → 用户干预（方向盘/油门/刹车）
    1.7.2.6.4  End maneuver            → 结束巡航

  1.7.2.7  [功能名] Park2Park
    1.7.2.7.1  Arrival Route Learning  → 到达路线学习
    1.7.2.7.2  Departure Route Learning → 离开路线学习
    1.7.2.7.3  Pre-Start P2P          → P2P 启动前
    1.7.2.7.4  Maneuvering P2P        → P2P 巡航
    1.7.2.7.5  Driving/Parking Handover → 行泊车切换场景
    1.7.2.7.6  Overlap Zone           → 重叠区域切换

  1.7.2.8  [功能名] Backend            → 云端地图下载/上传/版本检查
  1.7.2.9  [功能名] Hands Off Warning  → 手离方向盘检测与告警
  1.7.2.10 [功能名] Remote Control     → 远程控制（手机端）

  [根据 UC 情况可新增章节，如涉及行泊切换、多系统协同等场景]
```

## 追溯矩阵输出

完成 FR 拆解后，输出追溯矩阵。**只检查任务来源明确指定的 UC 条目**，未指定的不纳入范围：

| 1.5 章节 | 需求 type | 1.5 需求摘要 | 承接的 1.7.2 FR ID | 覆盖状态 |
|----------|----------|-------------|-------------------|---------|
| 1.5.2.2 | requirement | Set new recording map ID... | FR-03-01, FR-03-02 | ✅ |
| 1.5.6.11 | requirement | Obstacle evasion | FR-06-05, FR-06-06 | ✅ |
| 1.5.2.1 | note | Background learning remark | — | ⏭️ 跳过 |

- ✅ = requirement 类型，已有 FR 承接
- ⏭️ = 非 requirement 类型，跳过
- ❌ = requirement 类型，未覆盖，需补充 FR
