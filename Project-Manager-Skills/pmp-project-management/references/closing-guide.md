# 收尾阶段详细指导

## 目标

正式结束项目或阶段，有序移交成果，总结经验教训，释放资源。

## 核心过程

### 5.1 结束项目或阶段

### 1. 确认所有工作已完成
- 逐项对照范围基准的 WBS，确认所有工作包已交付
- 确认所有变更已实施并验证
- 确认所有未解决问题和缺陷已关闭或有明确处理方案
- 最终范围确认需获得发起人/客户书面签字

### 2. 获得正式验收
**验收流程：**
1. 编制验收清单（基于范围基准和验收标准）
2. 组织验收会议/评审会，逐项演示交付成果
3. 记录验收意见和遗留问题
4. 获得正式的验收签字/批准
5. 如有缺陷，编制缺陷清单，约定修复时限

### 3. 移交与运营过渡
- 确保运营/使用方已准备好接收：
  - 培训完成并考核通过
  - 操作文档和维护手册已交付
  - 运维支持渠道已建立（SLA 协议）
- 知识转移会议：将项目经验和技术知识传递给运营团队
- 移交签字确认

### 4. 总结经验教训
**经验教训总结会（Retro）议程：**

**1. 回顾目标（15分钟）**
- 项目原始目标是什么？实际达成情况如何？

**2. What went well 做得好的（20分钟）**
- 团队头脑风暴，记录所有正面经验
- 重点：哪些做法值得在未来项目中复用？

**3. What could be improved 可以改进的（20分钟）**
- 记录问题和不足
- 重点：根本原因是什么？下次如何避免？

**4. Action items 改进行动（15分钟）**
- 将改进建议转化为具体的行动项
- 指定责任人和完成时限

**经验教训登记册模板：**
| # | 日期 | 类别 | 描述 | 影响 | 建议 | 责任人 |
|---|------|------|------|------|------|--------|
| 1 | | 流程/技术/管理 | | | | |

**关键原则：**
- 营造安全的氛围，鼓励坦诚分享
- 不追究个人责任，聚焦系统和流程改进
- 经验教训必须具体可操作，避免空泛

### 5. 归档项目资产
**组织过程资产更新（OPA）：**
- 项目管理计划（最终版，含所有变更）
- 范围/进度/成本基准（最终版）
- 风险登记册（全部关闭）
- 变更日志（全部关闭）
- 问题日志（全部关闭）
- 经验教训登记册
- 所有状态报告
- 合同和采购文件（最终版）
- 质量测量结果

**可复用资产更新：**
- 更新项目模板（如发现模板可改进之处）
- 更新风险核对单（新增本项目的特有风险）
- 更新估算数据库（实际数据 vs 估算数据）
- 提交流程改进建议

### 6. 财务收尾
- 汇总所有实际成本并与预算对比
- 关闭所有采购订单和合同
- 结算供应商款项
- 归还未使用的预算和储备
- 编制最终成本报告

### 7. 团队收尾
- 完成团队成员绩效评估
- 正式释放团队成员（更新资源日历）
- 向团队表达感谢和认可
- 为团队成员提供职业发展反馈
- 举办项目庆祝活动（建议）

### 8. 行政收尾
- 确认所有合同义务已履行
- 更新组织数据库和系统
- 归档所有物理和电子文档
- 关闭项目编号/代码
- 通知所有相关方项目正式关闭

---

## 项目后评价（可选）

项目结束 3-6 个月后进行回顾性评价：
- 项目交付的业务价值是否实现？
- 运营方对交付成果的满意度如何？
- 经验教训是否被应用到其他项目中？
- 项目过程中识别的改进是否得到落实？

---

## 收尾阶段常见陷阱

| 陷阱 | 后果 | 预防措施 |
|------|------|---------|
| 急于结束，跳过验收 | 后续纠纷不断 | 严格按验收清单逐项签字 |
| 不做经验教训总结 | 组织无法从错误中学习 | 强制要求作为收尾必要条件 |
| 团队过早解散 | 文档缺失、知识流失 | 团队释放前完成所有归档 |
| 遗留缺陷不处理 | 运营方不满 | 遗留缺陷清单 + 修复计划 + 责任人 |
| 不更新组织过程资产 | 历史经验丢失 | 指定专人负责资产归档 |
